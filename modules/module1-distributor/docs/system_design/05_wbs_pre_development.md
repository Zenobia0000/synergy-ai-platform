# Synergy AI — 開發前交付物 WBS（工作分解結構）

> **版本:** v1.0 | **日期:** 2026-03-25 | **角色:** 產品經理視角
> **目的:** 明確列出開發前必須由各對口窗口準備的資料，確保開發團隊不會因為「缺資料」而卡住

---

## 讀這份文件前你需要知道的事

**這不是開發計畫，這是「開發之前」的準備清單。**

每個需求資料夾裡都有一份 `README.md`，告訴窗口：
1. 要準備什麼（交付物）
2. 為什麼需要這個（用途說明）
3. 格式長什麼樣（範本）
4. 什麼時候要給（截止日）
5. 給誰（收件人）

```
docs/system_design/requirements/
├── 01_domain_knowledge/       ← 領域專家提供
├── 02_questionnaire_design/   ← 領域專家 + 產品經理共同定義
├── 03_routing_rules/          ← 產品經理 + 資深經營者共同定義
├── 04_objection_knowledge_base/ ← 資深經營者提供
├── 05_training_program/       ← 培訓負責人提供
├── 06_content_templates/      ← 行銷窗口 + 資深經營者提供
├── 07_platform_accounts/      ← IT/營運窗口提供
└── 08_business_metrics/       ← 營運主管提供
```

### 0.1 領域知識盤點

📂 `requirements/01_domain_knowledge/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.1.1 | 產品目錄 | Excel/CSV | 所有產品名稱、SKU、適用情境、價格帶、組合包 |
| 0.1.2 | 客戶分群 | 文件 | 目前實際在用的客戶分類方式（不是理想的，是現在真的在用的） |
| 0.1.3 | 業務術語表 | 文件 | 內部術語（如「重生計畫」「MEGA」「上線」）的正式定義 |
| 0.1.4 | 現行作業流程 | 文件/影片 | 目前經營者從獲客到成交的實際操作步驟（含使用的工具） |

**為什麼需要**：開發團隊不懂你的業務。如果不給這些，系統欄位會對不上現實，做出來你不會用。

---

### 0.2 問卷設計

📂 `requirements/02_questionnaire_design/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.2.1 | 健康問卷欄位表 | Excel | 每個問題的題目、選項、是否必填、隱私等級 |
| 0.2.2 | FORM 背景欄位表 | Excel | Family/Occupation/Recreation/Money 每個欄位的問法與選項 |
| 0.2.3 | 問卷流程圖 | 圖 / 文件 | 哪些問題先問、什麼情況跳過、分段斷點在哪 |
| 0.2.4 | 隱私等級對照 | 文件 | 哪些問題消費者可以不答，不答時系統怎麼處理 |

**為什麼需要**：問卷是整套系統的資料入口。問錯問題 = 分流錯人 = 成交率不會上升。

---

### 0.3 分流規則

📂 `requirements/03_routing_rules/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.3.1 | 名單評分模型 | Excel | 評分維度、每個維度的權重、每個選項對應的分數 |
| 0.3.2 | 分流決策表 | Excel | 「當條件 A + B + C 成立時 → 執行動作 X」的完整對照表 |
| 0.3.3 | 例外情境 | 文件 | 規則衝突時的優先級、人工介入的條件 |

**為什麼需要**：沒有分流規則，AI 只會聊天，不會幫你賺錢。這是商業引擎的核心。

---

### 0.4 異議知識庫

📂 `requirements/04_objection_knowledge_base/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.4.1 | 異議分類 | Excel | 所有常見異議，依分類（價格/信任/時機/家人/產品）列出 |
| 0.4.2 | 回覆建議 | Excel | 每條異議至少 3 種回覆語氣（溫和/專業/直接） |
| 0.4.3 | 案例庫 | 文件 | 成功翻轉的案例 + 失敗收場的案例（用來訓練 AI 判斷） |

**為什麼需要**：新手經營者最怕被問倒。這個知識庫直接決定 AI 商談助手的品質。

---

### 0.5 培訓計畫

📂 `requirements/05_training_program/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.5.1 | 培訓課程大綱 | 文件 | MEGA 培訓的完整課程結構、每堂課主題 |
| 0.5.2 | 每週任務清單 | Excel | 新人每週要完成的具體任務（發幾篇文、聯繫幾人、參加什麼） |
| 0.5.3 | 評量標準 | 文件 | 怎麼判斷新人有沒有達標、什麼算「完成培訓」 |

**為什麼需要**：沒有結構化的培訓內容，系統只能追蹤「有沒有做」但不知道「要做什麼」。

---

### 0.6 內容模板素材

📂 `requirements/06_content_templates/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.6.1 | 社群貼文範例 | 文件 | FB/IG/X/LINE 各 10 篇高互動貼文，含圖文 |
| 0.6.2 | 私訊對話範例 | 文件 | 真實對話截圖或逐字稿，從初次接觸到邀約問卷 |
| 0.6.3 | 見證故事範例 | 文件 | 已發布的見證內容 10 則以上，含 B/A 照片 |
| 0.6.4 | 衛教內容清單 | Excel | 所有可用的衛教文章/影片/圖卡，含分類標籤 |
| 0.6.5 | 品牌語調指南 | 文件 | 品牌的說話風格、禁用詞、語氣規範 |

**為什麼需要**：AI 生成內容的品質取決於你給它的範例。垃圾進 = 垃圾出。

---

### 0.7 平台帳號與權限

📂 `requirements/07_platform_accounts/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.7.1 | 平台帳號清單 | Excel（加密） | FB Page ID、IG Professional ID、X Developer Account、LINE OA Channel |
| 0.7.2 | API Token/Key | 安全傳遞 | 各平台 API 存取 Token（不可放在文件裡，透過安全管道傳遞） |
| 0.7.3 | LINE OA 設定 | 文件 | LINE OA Messaging API 已開啟、Webhook URL 設定方式 |
| 0.7.4 | Google 帳號 | 安全傳遞 | 用於 Calendar 整合的 Google 帳號與 OAuth 設定 |

**為什麼需要**：沒有帳號權限，多平台串接完全無法測試。這是 Phase 5 的硬性前置。

### 0.8 營運指標基準

📂 `requirements/08_business_metrics/`

| 子項 | 交付物 | 格式 | 說明 |
| :--- | :--- | :--- | :--- |
| 0.8.1 | 現行轉化率 | Excel | 目前各階段的實際轉化率（有多少就給多少，沒有也要說沒有） |
| 0.8.2 | 目標值 | Excel | 導入系統後期望的目標值（用來設定儀表板預設目標） |

**為什麼需要**：沒有基準數據，儀表板做出來只是好看，不會幫你發現問題。

## 風險提醒

| 風險 | 影響 | 預防措施 |
| :--- | :--- | :--- |
| 領域專家沒空，0.1 延遲 | Phase 1 資料模型設計方向錯誤 | PM 先用訪談錄音 + 現有文件整理初版，再請專家驗證 |
| 問卷設計反覆修改 | Phase 2 延遲 | 限定最多 2 輪修改，第 3 輪後凍結版本先開發 |
| 分流規則窗口說「我不確定」 | 分流引擎做出來沒人敢用 | 先用最保守的規則上線，搭配人工 override，再根據數據調整 |
| 異議知識庫內容太少 | AI 商談助手品質低 | 設定最低門檻：至少 30 條異議 + 回覆才開始開發 |
| 平台帳號申請太慢 | Phase 5 無法測試 | Phase 1 就開始申請，不要等到需要時才辦 |

## 下一步行動

1. **PM 立刻做**：把這份 WBS 轉成窗口各自的 checklist，約第一輪訪談
