## ADR-001: 前端框架選型 — Vite + React

> **狀態:** 已接受 | **日期:** 2026-03-19 | **決策者:** kuanwei

### 1. 背景與問題

- **上下文**: 需要建立 Web 前端用於貼文管理與監控儀表板
- **問題**: 選擇前端框架以平衡開發效率與功能需求
- **驅動因素**: 快速開發、社群生態、與後端分離部署

### 2. 考量的選項

#### 選項一: Next.js
- **優點**: SSR/SSG 支援、API Routes 可合併後端、社群成熟
- **缺點**: 學習曲線較高、SSR 對本專案非必要、部署較複雜
- **成本/複雜度**: 中

#### 選項二: Vite + React (SPA)
- **優點**: 建置速度極快、配置簡單、與獨立後端自然分離
- **缺點**: 無 SSR（SEO 不重要）、需獨立後端
- **成本/複雜度**: 低

### 3. 決策

**選擇**: Vite + React (SPA)

**理由**: 本專案為個人工具，不需要 SEO 和 SSR。後端已決定使用 FastAPI (Python)，前後端自然分離。Vite 建置速度快，開發體驗佳。前端模板已用 shadcn/ui 產好，直接採用。

### 4. 後果

- **正面**: 開發速度快、部署簡單（靜態檔案）、前後端完全解耦
- **負面**: 首次載入需下載完整 JS bundle（可用 Code Splitting 緩解）
- **重新評估觸發**: 需要 SEO、多頁面 SSR 需求

## ADR-002: 後端框架選型 — FastAPI

> **狀態:** 已接受 | **日期:** 2026-03-19 | **決策者:** kuanwei

### 1. 背景與問題

- **上下文**: 需要 REST API 後端處理內容 CRUD、排程管理、與 n8n 整合
- **問題**: 選擇後端框架
- **驅動因素**: 開發效率、async 支援、型別安全、與 n8n webhook 整合

### 2. 考量的選項

#### 選項一: Express.js / Fastify (Node.js)
- **優點**: 與前端同語言、npm 生態豐富
- **缺點**: TypeScript 型別推導不如 Python type hints 直覺、ORM 選擇較分散
- **成本/複雜度**: 低

#### 選項二: FastAPI (Python)
- **優點**: 自動 OpenAPI 文檔、Pydantic 型別驗證、async/await 原生支援、SQLAlchemy 成熟
- **缺點**: 與前端不同語言、部署需 Python 環境
- **成本/複雜度**: 低

### 3. 決策

**選擇**: FastAPI (Python)

**理由**: FastAPI 提供自動 API 文檔、Pydantic 強型別驗證、async 支援。SQLAlchemy 是成熟的 ORM 方案。開發者對 Python 熟悉。

### 4. 後果

- **正面**: 自動產生 OpenAPI docs、Pydantic 驗證減少 bug、async 處理 webhook 回報效率高
- **負面**: 前後端不同語言、需管理 Python 虛擬環境
- **重新評估觸發**: 需要 WebSocket 密集應用、需與前端共享型別

## ADR-003: 資料庫選型 — PostgreSQL

> **狀態:** 已接受 | **日期:** 2026-03-19 | **決策者:** kuanwei

### 1. 背景與問題

- **上下文**: 需要持久化儲存貼文、發佈紀錄、監控數據
- **問題**: PostgreSQL vs SQLite
- **約束**: 需支援 JSON 欄位（recent_replies）、併發寫入（n8n 回報）

### 2. 考量的選項

#### 選項一: SQLite
- **優點**: 零配置、檔案式、簡單
- **缺點**: 併發寫入受限、不適合多程序存取、JSON 支援較弱
- **成本/複雜度**: 低

#### 選項二: PostgreSQL
- **優點**: 完整 JSON/JSONB 支援、併發能力強、擴展性佳
- **缺點**: 需額外部署資料庫服務
- **成本/複雜度**: 中

### 3. 決策

**選擇**: PostgreSQL

**理由**: n8n 與 FastAPI 可能同時寫入（發佈結果回報），需要良好的併發支援。monitor_data 的 recent_replies 需要 JSON 欄位。PostgreSQL 透過 Docker Compose 部署不增加太多複雜度。

### 4. 後果

- **正面**: 併發安全、JSON 查詢能力強、未來擴展無瓶頸
- **負面**: 需要 Docker 或本地安裝 PostgreSQL
- **重新評估觸發**: 無

## ADR-004: 自動化引擎 — n8n 本地自架

> **狀態:** 已接受 | **日期:** 2026-03-19 | **決策者:** kuanwei

### 1. 背景與問題

- **上下文**: 需要多平台發佈與定時監控的自動化引擎
- **問題**: n8n 雲端 vs 本地自架
- **驅動因素**: 成本、Webhook 延遲、資料隱私

### 2. 考量的選項

#### 選項一: n8n Cloud
- **優點**: 免運維、自動更新、穩定
- **缺點**: 月費成本、Webhook 需公網可達
- **成本/複雜度**: 中（月費）

#### 選項二: n8n 本地自架
- **優點**: 零成本、localhost 直接通訊、完全掌控
- **缺點**: 需自行維護、更新
- **成本/複雜度**: 低

### 3. 決策

**選擇**: n8n 本地自架

**理由**: 個人專案無需高可用，localhost 通訊延遲最低。n8n 透過 Docker Compose 與 PostgreSQL、FastAPI 統一管理。

### 4. 後果

- **正面**: 零成本、低延遲、資料完全本地
- **負面**: 需自行備份 workflow、更新版本
- **影響範圍**: Docker Compose 需包含 n8n 服務
- **重新評估觸發**: 需要 24/7 高可用、多設備存取

## ADR-005: 認證機制 — MVP 預留，不實作

> **狀態:** 已接受 | **日期:** 2026-03-19 | **決策者:** kuanwei

### 1. 背景與問題

- **上下文**: 專案需要使用者認證機制
- **問題**: MVP 階段是否實作完整認證
- **約束**: 單人使用、時程有限

### 2. 決策

**選擇**: MVP 不實作認證，但架構預留 JWT middleware 接口

**理由**: 單人使用且在本地運行，認證非必要。FastAPI middleware 架構天然支援後續插入認證，預留 `SECRET_KEY` 和 `python-jose` 依賴即可。

### 3. 後果

- **正面**: 減少 MVP 開發量約 1-2 天
- **負面**: 本地部署期間 API 無認證保護
- **重新評估觸發**: 部署至公網、多人使用

## ADR-006: 前後端通訊 — REST + 輪詢

> **狀態:** 已接受 | **日期:** 2026-03-19 | **決策者:** kuanwei

### 1. 背景與問題

- **上下文**: 前端需即時顯示發佈狀態更新
- **問題**: 如何實現即時通訊（WebSocket vs SSE vs 輪詢）

### 2. 考量的選項

#### 選項一: WebSocket
- **優點**: 全雙工即時通訊
- **缺點**: 實作複雜、需要連線管理
- **成本/複雜度**: 高

#### 選項二: Server-Sent Events (SSE)
- **優點**: 伺服器推送、HTTP 原生、簡單
- **缺點**: 單向、需要保持長連線
- **成本/複雜度**: 中

#### 選項三: 短輪詢 (Polling)
- **優點**: 實作最簡單、無需額外基礎設施
- **缺點**: 有延遲（取決於輪詢間隔）、增加 API 請求量
- **成本/複雜度**: 低

### 3. 決策

**選擇**: REST API + React Query 短輪詢（refetchInterval）

**理由**: 發佈頻率低（每天數次），不需要即時到毫秒級。React Query 的 refetchInterval 可以每 5-10 秒輪詢一次，滿足「即時」需求且實作最簡單。未來可升級為 SSE。

### 4. 後果

- **正面**: 實作簡單、不需額外連線管理
- **負面**: 狀態更新有 5-10 秒延遲
- **重新評估觸發**: 需要毫秒級即時更新、API 負載過高
