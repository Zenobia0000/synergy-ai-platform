# 實作計畫：解析 rawdata 為 JSON schema

## 目標

將 4 份異質 rawdata（Apps Script docx、2 份 xlsx 問卷、product_catalog.xlsx、產品圖片連結 docx）解析並**正規化**為 3 份 JSON：`questionnaire.json`、`products.json`、`product_rules.json`，放置於 `data/schemas/`，作為後續 Pydantic schemas（2.2）、LLM prompts（3.x）、前端問卷表單（5.3）共用的**單一資料源（Single Source of Truth）**。

## 關鍵決策（使用者確認）

| 決策 | 選擇 | 理由 |
|---|---|---|
| 問卷主版本 | `諮詢問卷_含產品建議.xlsx` | 欄位最全、含穩定 `欄位代碼`、含產品對應欄 |
| docx JS 源碼解析 | 正則擷取（不做 AST） | 結構穩定、正則足以覆蓋 `addXxxItem().setTitle()` |
| 圖片連結對應 | 精確 + 正規化 + `difflib` ≥ 0.85 三層匹配 | 容錯名稱空白/符號差異 |
| 未匹配圖片策略 | 寫入 `unmatched_images.json`，image_url 設 null，**不阻斷** | 避免單筆名稱問題導致整體失敗 |
| 輸出位置 | `data/schemas/`（專案根下） | 前後端共用、最鬆耦合 |

## 技術依賴

- Python 3.12 + uv（openpyxl、python-docx、pydantic 已安裝）
- 新增依賴：無（fuzzy 匹配用 stdlib `difflib.SequenceMatcher`）

- 中文檔名用 `pathlib.Path`，不字串拼接
- parser 獨立成 `backend/app/services/parsers/` 子套件，便於測試

## 風險與緩解

| 風險 | 影響 | 緩解 |
|---|---|---|
| 問卷三版本不一致 | HIGH | 主 xlsx 為準，docx 僅驗證，產 `_consistency_report.md`，差異 > 20% 提示人工 |
| docx 中文檔名含全形 `｜` | MEDIUM | 全程 `pathlib.Path`，CLI 用 `--input-dir` + glob，不硬編檔名 |
| 圖片名稱與 SKU 對不上 | HIGH | 三層匹配 + `unmatched_images.json` 人工清單，不阻斷 |
| 產品目錄有組合包、重複、空列 | MEDIUM | 空列跳過、重複取最後 + warning、組合包放 `bundle_contents` |
| 條件顯示自然語言難解析 | MEDIUM | 先實作 `若 X == Y` / `若 X >= Y`，無法解析保留 `condition_raw` |
| 價格欄格式雜亂 | LOW | 抽數字，無法解析 `price=null`、保留 `price_display` |
| rawdata 變動 | MEDIUM | `_meta.source_sha256` 可讓 CI 偵測 |
| LLM 語義不足 | MEDIUM | schema 保留 `label` 中文原文、`help_text` |
