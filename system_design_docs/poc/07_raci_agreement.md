## 1. 合作模式定義

### 1.1 共創原則

本 POC 採**合夥共創型**合作模式，客戶方與開發方共同投入、共同承擔成果，而非單純的甲乙方委託關係。以下為核心共識：

| 原則 | 說明 |
| :--- | :--- |
| **平等發聲** | 任何一方均可對需求、設計、優先序提出意見；意見不因角色而有輕重之別 |
| **共同擁有成果** | 模組功能、使用者體驗、商業效益均為雙方共同交付，不單獨歸功或歸咎 |
| **互補專業** | 客戶方掌握領域知識與商業判斷；開發方掌握技術實現；雙方各自尊重對方的專業邊界 |
| **透明進度** | 每週同步進度、障礙與決策需求；不累積問題、不在最後才揭露風險 |
| **快速迭代** | 以 Phase Gate 為節奏，每個里程碑後共同回顧並調整方向 |

### 1.2 決策機制

```
優先以共識決策（Consensus First）：
  ├── 雙方同意 → 執行
  ├── 有歧見 → 討論，嘗試整合觀點
  └── 僵局時：
        ├── 業務範疇 / 需求優先序 → 客戶方最終決定
        └── 技術方案 / 架構選型 → 開發方最終決定
```

> **原則：** 決策後雙方均接受並全力執行，不在執行中反複。任何決策均應書面記錄於 Slack 頻道或週會紀錄。

### 1.3 透明義務

- **開發方**：每週一同步本週進度、下週計畫、目前風險；不隱瞞技術債或品質問題
- **客戶方**：提前告知業務需求異動、交付物可能延遲；不在最後一刻才反映
- **共同**：任何影響時程超過 1 天的事件，24 小時內主動通知對方

## 4. 溝通節奏

### 4.1 定期同步

| 會議類型 | 頻率 | 時長 | 參與者 | 目的 |
| :--- | :--- | :--- | :--- | :--- |
| **週會（Weekly Sync）** | 每週一 | 30 分鐘 | 客戶方代表 + PM/QA（+ 按需開發成員） | 進度回顧、障礙揭露、本週決策事項 |
| **Phase Gate Review** | 每個里程碑（M1～M5）| 60 分鐘 | 雙方全員 | 階段成果展示、驗收確認、下階段目標對齊 |
| **Ad-hoc 會議** | 按需 | 30 分鐘 | 相關人員 | 緊急決策、需求澄清 |

### 4.2 非同步溝通

| 管道 | 用途 | 回應時限 |
| :--- | :--- | :--- |
| **Slack / LINE 頻道** | 日常問題、資訊分享、快速確認 | 工作日 4 小時內 |
| **週會紀錄（共用文件）** | 決策記錄、行動事項追蹤 | 會後 24 小時內更新 |
| **變更請求表單** | 正式範疇異動申請 | 3 個工作日內回應 |

### 4.3 Ad-hoc 會議規則

- 任何一方均可提出，須提前 **24 小時**通知
- 發起方提供議程（1-3 個議題），確保會議聚焦
- 若非緊急，優先在下次週會處理以減少打斷

## 5. 變更管理

### 5.1 範疇變更流程

```
1. 發起
   任何一方識別需求異動，填寫「變更請求單」
   （填寫人、變更內容、原因、預估影響）
     ↓
2. 影響評估（3 個工作日內）
   雙方各自評估：時程影響 / 成本影響 / 品質影響
     ↓
3. 聯合決策
   ├── 批准（Approve）→ 更新 WBS、Gantt、通知所有相關人
   ├── 拒絕（Reject）→ 記錄原因，維持原計畫
   └── 延後（Defer）→ 列入 POC 後待辦，不影響本期時程
     ↓
4. 執行（僅限批准項目）
   PM/QA 更新 WBS 與甘特圖，同步通知全員
```

### 5.2 變更授權層級

| 變更規模 | 定義 | 審批人 |
| :--- | :--- | :--- |
| **小型變更** | 影響 < 2 人日，不改變里程碑日期 | PM/QA 可直接批准 |
| **大型變更** | 影響 ≥ 2 人日，或影響里程碑日期 | 需客戶方 + 開發方雙方共同簽核 |

### 5.3 鐵律

- **禁止口頭範疇變更。** 所有範疇異動須書面記錄（Slack 訊息截圖亦可），不接受「你說過要做 X」的口頭聲明。
- 若口頭討論達成共識，由 PM/QA 在 24 小時內以書面確認，雙方確認無誤後方才生效。
- 變更紀錄保存於共用文件，作為日後爭議的唯一依據。

## 6. 智財與資料歸屬

| 資產類型 | 歸屬方 | 說明 |
| :--- | :--- | :--- |
| **原始碼（Source Code）** | 雙方共同擁有 | POC 期間產出的所有程式碼由雙方共同持有；後續商業化另行協議 |
| **業務資料與內容** | **客戶方（kuanwei / Synergy）** | 客戶提供的問卷題目、分流規則、培訓素材、品牌內容等業務資料，完整歸屬客戶方 |
| **AI Prompt 與模型設定** | 雙方共同擁有 | POC 期間共同開發的 prompt 設計、模型調校策略，雙方均可使用 |
| **UI 設計資產** | 雙方共同擁有 | 設計 token、元件庫由雙方共同持有 |
| **開源依賴套件** | 各套件授權 | 依 MIT、Apache 2.0 等個別授權管理，雙方均須遵守 |
| **使用者個資** | **客戶方** | 問卷填寫者等終端使用者資料，由客戶方依法負責保管與合規 |

> **保密義務：** 雙方均不得將對方的商業機密、技術細節或使用者資料揭露給未授權的第三方。
