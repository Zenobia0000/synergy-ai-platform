## 1. 商業假設（Business Hypothesis）

> 我們相信，**將健康管理事業的核心工作流程（行銷內容產製 → 問卷篩選潛客 → 團隊目標管理）全面數位化並導入 AI 輔助**，將能**提升客戶轉換率並實現團隊複製擴張**，因為目前這三個環節高度仰賴個人經驗與手動操作，導致品質不穩定、時間成本高，且難以橫向規模化。

**核心問題陳述：**

| 痛點 | 現況 | 目標
|---|---|---|
| 行銷內容 | 手動撰寫，品質與產量受限於個人 | AI 批量產製，維持或超越人工品質 |
| 潛客評估 | 依賴業務主觀判斷 | 結構化問卷 + AI 分析，量化篩選準確度 |
| 團隊管理 | 口頭或手動追蹤目標，無法橫向複製 | 數位儀表板，任何層級主管皆可即時掌握 |

---

## 2. POC 目標（Objectives）

本次 POC 旨在驗證以下三項核心假設：

| # | 假設 | 驗證方法 | 最低通過標準 |
|---|---|---|---|
| H1 | AI 生成內容可達到或超越人工撰寫的社群貼文品質 | 盲測評分（業務團隊 5 人）、平台互動率追蹤 | 盲測評分 ≥ 人工分數 80%，產出時間縮短 ≥ 50% |
| H2 | 結構化問卷 + AI 分析可改善潛客資格篩選準確度 | 對比舊流程的轉換率，Lead Score 分佈分析 | 問卷完成率 ≥ 60%，Lead Score 準確率（人工複核）≥ 75% |
| H3 | 數位化團隊管理儀表板可取代手動追蹤並提升目標達成率 | 主管採用率、月目標達成率變化 | 主管主動使用率 ≥ 70%，目標設定完成率 ≥ 80% |

詳細指標定義見 `02_success_metrics.md`。

### 3.2 本次 POC 排除（Out-of-Scope）

| 模組 | 排除原因 |
|---|---|
| M3 商談轉化 | 依賴 M2 穩定資料流，排在正式版 |
| M4 客戶管理 | CRM 整合複雜度高，列為下一階段 |
| M5 成果與轉介紹 | 需 M3/M4 資料基礎，暫緩 |

### 3.3 延後討論（Deferred）

| 項目 | 理由 | 預計時間點 |
|---|---|---|
| 多語系（英文／日文）| POC 聚焦繁中市場 | 正式版 Phase 2 |
| 行動 App（iOS/Android）| Web-first 驗證需求優先 | 正式版 Phase 3 |
| 自動化 CRM 雙向同步 | API 整合工時大，POC 改人工匯入 | M4 開發後 |

## 4. 成功定義（Success Definition）

POC 結束時，滿足以下條件視為「成功可交付」：

| 層級 | 標準 |
|---|---|
| 功能可用性 | 所有 P0 功能完成且通過 UAT；P1 功能完成度 ≥ 80% |
| 假設驗證 | H1、H2、H3 至少 2 項達最低通過標準 |
| 技術穩定性 | API 錯誤率 < 1%，平均回應時間 < 3 秒 |
| 使用者體驗 | 客戶方真實用戶 NPS ≥ 7 / 10 |

完整量化指標定義詳見 [`02_success_metrics.md`](./02_success_metrics.md)。

## 7. Go / No-Go 決策閘門（End-of-POC Gate）

POC 結束時（2026-07-27）進行正式評估會議，依以下決策樹決定後續方向：

```
POC 結束評估
    │
    ├─ 所有 P0 功能通過 UAT，且 ≥ 2 項假設達標
    │       └─ ✅ GO → 啟動正式版開發
    │              - 新增 M3 商談轉化、M4 客戶管理、M5 成果與轉介紹
    │              - 重新議定正式版合約與時程
    │
    ├─ 部分功能通過，假設驗證結果不一
    │       └─ 🔄 PIVOT → 調整範疇或技術路線
    │              - 選項 A：縮減正式版範疇，聚焦已驗證模組
    │              - 選項 B：延長 POC 4–6 週，補強薄弱假設
    │              - 選項 C：替換技術方案（如換模型、調整架構）
    │
    └─ 核心假設根本不成立（如 AI 品質無法被接受）
            └─ ❌ KILL → 終止專案
                   - 交付技術報告與學習文件
                   - 討論退場條款
```

**評估會議出席：** 雙方決策負責人、開發 lead、PM/QA
