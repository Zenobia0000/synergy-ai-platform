## 1. 技術棧確認


| 技術                         | 狀態  | 使用模組       | 選用原因                                               | 風險等級 |
| -------------------------- | --- | ---------- | -------------------------------------------------- | ---- |
| FastAPI (Python)           | 已有  | M1, M2, M6 | 已驗證於 M2 POC；非同步支援佳；OpenAPI 自動生成                    | 低    |
| PostgreSQL                 | 已有  | M1, M2, M6 | 結構化資料，跨模組共用；M1 已在使用                                | 低    |
| Vite + React 18            | 已有  | M1         | M1 既有前端；shadcn/ui 相依 React 18                      | 低    |
| Vite + React 19            | 已有  | M2, M6     | M2 POC 已驗證；M6 對齊 M2 技術棧                            | 低    |
| LiteLLM + Gemini 2.5 Flash | 已有  | M1, M2     | 經濟實惠（Flash 方案）；M2 POC 已驗證文字生成品質；LiteLLM 提供多模型切換彈性  | 低    |
| n8n (self-hosted)          | 已有  | M1         | 工作流自動化；貼文排程分發；無授權費                                 | 低    |
| Tailwind CSS v4            | 已有  | M2, M6     | 快速 UI 開發；M2 已驗證；M6 新建對齊 M2                         | 低    |
| shadcn/ui                  | 已有  | M1         | M1 既有元件庫；豐富可組合元件                                   | 低    |
| uv (Python 套件管理)           | 已有  | M2, M6     | 快速依賴管理；比 pip 顯著更快；M2 已驗證                           | 低    |
| Image generation API       | 新增  | M1         | 貼文圖片生成需求；需評估 Gemini Imagen vs Stable Diffusion API | 中    |
| MEGA API / CSV 匯入          | 新增  | M6         | 培訓與目標數據主要來源；API 可存取性未經驗證                           | 高    |
| Email / LINE 通知            | 新增  | M6         | 績效告警推送；LINE Messaging API 有免費層                     | 低    |
| 圖表套件（Recharts / Chart.js）  | 新增  | M6         | 績效看板視覺化；Recharts 與 React 生態整合較佳                    | 低    |


**備註 — React 版本分歧：** M1 使用 React 18，M2 使用 React 19。M6 建議採用 React 19 + Tailwind v4 以對齊 M2，降低未來跨模組元件整合成本。三模組前端在 POC 階段維持各自獨立，不強制統一。

## 4. 資料可行性


| 資料類型              | 來源            | 格式          | 預估品質           | 取得方式           | 隱私合規需求                |
| ----------------- | ------------- | ----------- | -------------- | -------------- | --------------------- |
| 健康問卷設計（題目與選項）     | 客戶方提供         | Excel       | 中（需結構化轉換）      | 客戶交付文件         | 需依資料分級處理，含個人健康資訊      |
| 產品目錄（品項、分類、定價）    | 客戶方提供         | Excel / CSV | 高              | 客戶交付文件         | 無敏感資料                 |
| 分流規則（問卷結果 → 推薦邏輯） | 客戶方 + PM 共同定義 | 結構化規則文件     | 低（需多輪迭代確認）     | 工作坊共同定義        | N/A                   |
| 社群內容模板（文案 + 圖片）   | 客戶方提供         | 圖文檔         | 中              | 客戶交付           | 需確認版權與脫敏需求            |
| MEGA 培訓資料         | MEGA 內部系統     | 未知          | 未知             | API 串接或 CSV 匯出 | 待 Spike S1 確認；可能含成員個資 |

**資料整備行動項目：**

- 分流規則需於 Phase 1 W1 完成第一版共識，作為 M2 AI 分析的 ground truth
- MEGA 資料格式待 Spike S1 確認後，規劃匯入 pipeline

## 5. 整合複雜度評估


| 整合點                  | 涉及模組       | 複雜度 | 說明                                                 |
| -------------------- | ---------- | --- | -------------------------------------------------- |
| 問卷潛客名單 → 團隊分配        | M2 → M6    | 中   | Lead 資料需跨模組流動；需定義共用資料結構與事件觸發機制                     |
| AI 生成內容 → 排程發布       | M1 內部      | 低   | n8n 已建基礎；新增 LLM 生成節點即可                             |
| 問卷評估結果 → 行銷內容個人化     | M2 → M1    | 低   | Phase 4 整合；POC 階段各自獨立                              |
| 共用認證（登入 / RBAC）      | M1, M2, M6 | 中   | JWT + role-based access；需統一 token 設計；Spike S4 先行確認 |
| 共用 PostgreSQL schema | M1, M2, M6 | 高   | 三模組共用同一 DB；schema 設計需跨模組協調，避免命名衝突與循環依賴             |
| 前端統一入口（路由整合）         | M1, M2, M6 | 中   | POC 階段各模組可獨立 port；正式版需統一入口，涉及前端路由設計與 React 版本整合    |


**高複雜度整合緩解策略：** 共用 DB 於 Phase 0 統一設計 ERD，以模組前綴（`m1_`, `m2_`, `m6_`）區隔命名空間，減少衝突機率。

## 6. 技術風險 Top 5


| #   | 風險描述                          | 影響範圍                       | 發生機率 | 緩解策略                                                 | Contingency                             | 負責人     |
| --- | ----------------------------- | -------------------------- | ---- | ---------------------------------------------------- | --------------------------------------- | ------- |
| T1  | MEGA API 無法存取或格式不符            | Module 6 核心功能延誤 1–2 週      | 高    | Phase 0 W1 即執行 Spike S1；提早接觸 MEGA 管理員                | 改用 CSV 手動匯入作為 fallback，功能降級但不卡關         | AI 工程師  |
| T2  | Meta 平台（FB/IG）App 審核延遲        | Module 1 發布功能無法上線          | 中    | 沿用既有已審核帳號；提早提交審核申請（至少提前 2 週）                         | Mock 發布模式 + 手動貼文驗證 UAT                  | 全端工程師 A |
| T3  | LLM 幻覺或輸出品質不穩                 | M2 問卷分級誤判；M1 內容品質低落        | 中    | 規則前置過濾（輸入 schema 驗證）+ 人工複核 UI；Spike S3 建立品質 baseline | 降級為純規則引擎（無 AI），保留 AI 為選配                | AI 工程師  |
| T4  | React 18 / 19 版本不一致，元件無法跨模組共用 | 跨模組元件複用困難，重複開發成本           | 低    | M6 統一採用 React 19；M1 於 POC 後評估升級                      | 各模組維持各自獨立前端，不強制共用元件                     | 全端工程師 B |
| T5  | 共用 DB schema 跨模組衝突            | Phase 3–4 整合階段需大量修改，影響交付時程 | 中    | Phase 0 統一設計 ERD，定義模組命名規範與共用表格（如 `users`、`teams`）    | 各模組使用獨立 schema + 整合層 View，犧牲部分查詢效率換取隔離性 | 全端工程師 A |

## 附錄：技術決策摘要


| 決策          | 選擇                     | 理由                                                  |
| ----------- | ---------------------- | --------------------------------------------------- |
| M6 前端框架     | React 19 + Tailwind v4 | 對齊 M2，降低技術碎片化                                       |
| M6 圖表套件     | Recharts（建議）           | React 生態原生整合；輕量；可替換為 Chart.js                       |
| LLM 閘道      | LiteLLM                | 已驗證；多模型切換彈性高；避免廠商鎖定                                 |
| Python 套件管理 | uv（M2, M6）/ pip（M1 既有） | M1 維持現狀降低遷移風險；新模組統一 uv                              |
| 圖片生成        | 待 Spike S2 決定          | Gemini Imagen（若 Preview 品質可用）或 Stable Diffusion API |
